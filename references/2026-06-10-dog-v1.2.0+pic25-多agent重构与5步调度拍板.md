## 🔴 第一次纠错（"你看到子 agent 介入吗"）

> "我记得这个skill的设计是有子agent介入的，但是我怎么没有看到你调用子agent来执行任务啊？"

> "请你重点检查一下这方面的问题呢"

### 我犯的错误

**主 agent 凭印象算 = 上下文污染**。具体表现：
- ❶ 算旁白时长 = 我在主 agent 里跑 Python 算（凭印象 1.4 词/秒兜底公式）= **没调 B 子 agent**
- ❷ 算 Clip 时长档位 = 我在主 agent 里挑（5/6/7/14s）= **没调 BC 子 agent**
- ❸ 写分镜脚本 = 我在主 agent 里拼 = **没调 C 子 agent**
- ❹ 写 prompt 草稿 = 我在主 agent 里拼 v15 6 段 = **没调 P 子 agent**
- ❺ 跑视频 = 我用 subprocess.Popen 调 seedance.py = **没调 D 子 agent**

= **5 件事全部在主 agent 里跑** = **子 agent 架构 = 名存实亡** = **Pic 5 实战沉淀的"主 agent 同时承担 4 件事 = 上下文污染"教训**根本没遵守。

### v1.0.0 旧架构的设计初衷

`references/v1.0-refactor-plan-multi-agent-orchestration.md` 早就定义了 4 个子 agent：
- A 风格识别（输入简介+图 → 输出调性+风格锚定词）
- B 旁白量化（输入旁白 → 输出朗读时长+复杂度）
- C 分镜设计（输入 A+B → 输出节奏公式+镜头表+v15 prompt 草稿）
- D 视频执行（输入 C → 输出 task_id+mp4+抽帧报告）

但**我**一直把 A+B+C+D **全部压缩在主 agent 上下文里跑** = 完全违背了多 agent 架构。

## 🟡 用户给的新流程（5 步 + 4 子 agent 建议）

### 用户原话

> "你看我们现在的工作流程是：
> 1. 我把素材包发给你，把旁白内容发给你
> 2. 主agent直接来做视觉识别，并把视觉识别的信息传递给子agent
> 3. 子agent结合图片和原始旁白的内容，对Clip的时长和节奏进行划分
> 4. 划分完成之后交给分镜设计的子agent，让他根据前面的内容来设计合适的分镜脚本
> 5. 分镜脚本设计完成之后，再交给提示词编写的子Agent来专门写视频提示词
> 你觉得这个路径是否比现在的流程更好？"

### 我提了 3 个方案

| 方案 | 架构 | 适用 |
|------|------|------|
| A · 5 子 agent 严格独立 | A 视觉主 agent / B+R 节奏 / C 分镜 / P prompt / D 执行 | 正式绘本（精度高）|
| B · 3 子 agent 精简 | A 视觉主 agent / BC 综合 / P prompt / D 执行 | 快速迭代（成本低）|
| C · 现有 4 子 agent + 视觉识别归主 agent | 4 子 agent 不变 + 主 agent 跳视觉识别 | 最小改动 |

### 用户拍板的修正（关键）

> "我看到你写的那个方案里面，风格确定还需要有一个agent去做，这个是不合适的。
> 因为主agent已经做了风格识别（也就是视觉识别），视觉识别里面已经会锁定风格了，就不需要去专门识别图片，再去做风格锁定，对吧？"

**关键洞察**：
- **不需要单独的"风格识别子 agent"** —— A 子 agent 是 v1.0.0 旧架构遗留
- **风格 = 主 agent native vision 视觉识别的一部分**
- 视觉识别一步到位输出 3 件事：① 画面描述 ② 风格锚定 ③ 5 种画面状态

## 🟢 v1.2.0+pic25 拍板（用户两次纠错 → 拍板 4 子 agent 架构）

### 4 子 agent 最终状态

| Agent | 职责 | 状态 |
|---|---|---|
| ~~A · 风格识别~~ | ~~调性 + 节奏倾向 + 风格锚定词~~ | ❌ **v1.2.0+pic25 已废 · 删除** |
| **B · 旁白量化** | 朗读时长 + 复杂度 + 静默推荐 | ✅ 保留 |
| **BC · 节奏/时长划分**（新增）| 时长档位 + 节奏公式 + 镜头数 | ✅ **新增** |
| **C · 分镜设计**（重定义）| 末帧策略 + 镜头描述（不带 prompt） | ✅ **重写** |
| **P · 提示词编写**（新增）| 完整 v15 范式 prompt 草稿 | ✅ **新增** |
| **D · 视频执行** | seedance 跑 + ffmpeg 抽帧 + vision 自检 | ✅ 保留 |

### 6 步调度（v1.2.0+pic25）

```
Step 0 接收需求
Step 1 启动前 7 必问
Step 2 主 agent native vision 视觉识别（不调子 agent）
Step 3 调 B 旁白量化 + BC 节奏/时长（并行）
Step 4 调 C 分镜设计（不拼 prompt）+ P 提示词编写（独立·串行）
Step 5 调 D 视频执行（3 并发）
Step 6 汇总 + 4 步对账 + 发飞书必附视频附件
```

### 拍板原则

1. **视觉识别 = 主 agent native vision**（**不**调子 agent · 避免上下文污染）
2. **时长/节奏/分镜/prompt = 各自独立子 agent**（**不**在主 agent 凭印象算）
3. **每个子 agent = 单一职责**（Pic 5 实战沉淀：v0.7.1+pic7 时代主 agent 同时承担 4 件事 = 上下文污染）

## 🎯 元教训（4 问审查 = 铁律级）

### 元教训 #1：多 agent 架构必须真调

**违反 = 必然上下文污染**（Pic 5 实战沉淀 + v0.7.1+pic7 时代翻车教训）

- ❌ 主 agent 凭印象算 = 拼 prompt = 写分镜 = 跑视频
- ✅ 视觉识别 = 主 agent native vision（**不**调子 agent）
- ✅ 时长/节奏/分镜/prompt = **必**调子 agent
- ✅ 每个子 agent = 单一职责

### 元教训 #2：架构变更必须 5 步走（删除 A 的子 agent 教训）

**违反 = 重复 agent 浪费上下文**（用户纠错"风格已经在视觉识别里 = 不需要单独 agent"）

- ❌ 拆得太细（每个动作都一个 agent）= 调度开销大
- ❌ 拆得太粗（一个 agent 干 4 件事）= 上下文污染
- ✅ 拆 C 子 agent = 纯分镜脚本（不拼 prompt）
- ✅ 拆 P 子 agent = 专门写 prompt
- ✅ 用户拍板的粒度 = 1 个萝卜 1 个坑 = 各自 prompt 模板更专

### 元教训 #3：v22 撞 100% = 幻觉天花板（达尔文 test-prompts 真实性审查）

**违反 = 虚假命中 = 达尔文失去价值**（v1.2.0+pic22 撞 100% 撞成幻觉天花板教训）

- ❌ test-prompts 引用已废 # 编号铁律（#32 #33 #36 #38）= 虚假命中
- ❌ test-prompts 引用不存在的 references 文件 = 虚假命中
- ✅ test-prompts.json 全文搜覆盖率 = 真实覆盖率
- ✅ v1.2.0+pic23 真实覆盖率 11/11 = 100%（重写后）
- ✅ v1.2.0+pic25 真实覆盖率 20/20 = 100%（v25 关键改动 8/8 全覆盖）
