## Part 1 · 铁律维护工作流

### 1.1 触发场景

- 用户说"铁律太多了" / "铁律定义有问题" / "这 2 条铁律适用场景不对"
- 准备加新铁律前
- 整理铁律清单时
- 维护 picturebook-video / seedance2.0-tool / 其他相关 skill 仓

### 1.2 完整工作流（7 步 · 2026-06-10 实战沉淀）

```
Step 1 · 压缩上下文
  ↓ （用 session_search 找历史关键判断 · 或直接读 SKILL.md 当前状态）
Step 2 · 拉出当前铁律清单（脚本正则 / 人眼读）
  ↓
Step 3 · 用户纠错点分类
  ├─ 红线缺失 → 加
  ├─ 红线重复 → 合并
  ├─ 红线误场景 → 改适用场景描述
  └─ 流程/方法/技巧混进铁律表 → 降级到 references
Step 4 · 元教训判定（4 问）
  1. 违反 = 必然翻车吗？
  2. 能合并到其他铁律吗？
  3. 能降级到 references 或 SKILL.md 正文吗？
  4. 有官方文档原话支持吗？
  ↓
Step 5 · 修正 SKILL.md 铁律表（短句 + 指向 references）
  ├─ 删去括号内长解释（解释性文字 → references）
  ├─ 分组（范式路由 / 官方硬约束 / 主 agent 行为 / 等等）
  └─ 标题条数 = 实际条数
Step 6 · 同步 Notion 页面（用 execute_code + python urllib + PATCH 逐 block 改）
  ↓
Step 7 · 报用户查看 · 验证一致性
```

### 1.3 元教训升铁律通道（用户 2026-06-10 新维度）

**用户原话**：「**降级的元教训可以作为铁律**」

**两类铁律**（同等地位）：

| 类型 | 来源 | 例子 |
|---|---|---|
| **红线铁律** | 官方文档/物理约束/用户底线 | #90 时长 4-15s · #72 整数 duration · ⭐官方§4 1 镜头 1 运镜 |
| **元教训铁律** | 修复翻车原则/行为准则 | #99 修复翻车 ≠ 加新东西（减法 > 加法）· #87 主 agent 不主动抽帧自检 |

**判定口诀**：

- "违反会破坏产品/用户体验" = 元教训铁律
- "违反会让 agent 重复犯同样错" = 元教训铁律
- "违反必翻车（物理/官方层）" = 红线铁律
- "违反不翻车（流程/方法层）" = 降级到 references

**降级反向也成立**：用户判定"这条不够格铁律" → 降级到 `references/元教训-YYYY-MM.md` 或 `references/失败记录-YYYY-MM.md`。

### 1.4 加新铁律必跑流程（4 问 + 4 必查）

1. **自问"违反会怎样"**——不能回答 = 不是铁律
2. **必查是否已有同类铁律**——重复 = 不加
3. **必查官方文档原话支持**（参见 `seedance2.0-tool/references/seedance-official-docs-research-2026-06-10.md`）
4. **主 agent 主动征求用户"算不算铁律"再写入**（不擅自加）

### 1.5 维护 references 必跑流程（4 问）

1. **必问"是不是该 patch 现有 references"**——已有 `v7-vs-v15-paradigm-routing.md` / `leading-reading-4clip-pattern.md` / `长旁白拆分规范-v15.1.md` 三大主题文档，新文档是否应作为它们的扩展/合并而非独立文件？
2. **必问"新文档是不是顶层入口"**——是顶层入口（如决策树/路由总览）→ 新建独立文件；是子规则/案例 → 写到对应 references 末尾章节
3. **新建独立文件必含指针区**——每章/每节末尾链回 3 大主题文档（不孤立）
4. **SKILL.md §"详细规范位置"必加一行**——否则下次 agent 找不到

### 1.6 压缩上下文实战

**session_search 通常无结果**（hermes 当前不会自动索引老会话），**替代方法**：

- 直接 `skill_view` 当前 SKILL.md（最新状态）
- 找 1-2 个 references 验证关键判断（不要全读）
- 用户原话/纠错点 = 直接从当前 session context 取

### 1.7 Notion 同步工具函数

**完整脚本**（已实测成功）：

```python
import urllib.request as ur, json, os
NOTION_API_KEY = os.popen("grep '^NOTION_API_KEY=' /home/luo/.hermes/.env | cut -d'=' -f2").read().strip()
page_id = "<your-page-id>"  # 36aa3f9269aa80e5af31c731a5b6afee
hdr = {
    "Authorization": f"Bearer {NOTION_API_KEY}",
    "Notion-Version": "2025-09-03",
    "Content-Type": "application/json"
}

# 1) 读所有 block
req = ur.Request(f"https://api.notion.com/v1/blocks/{page_id}/children?page_size=100", headers=hdr)
with ur.urlopen(req) as r:
    blocks = json.loads(r.read())["results"]

# 2) 删 block
req = ur.Request(f"https://api.notion.com/v1/blocks/{block_id}", method="DELETE", headers=hdr)
with ur.urlopen(req) as r:
    pass  # status 200

# 3) 改 block (PATCH)
body = json.dumps({
    "callout": {
        "rich_text": [{"type": "text", "text": {"content": "新文"}}],
        "icon": {"type": "emoji", "emoji": "🔥"}
    }
}).encode()
req = ur.Request(f"https://api.notion.com/v1/blocks/{block_id}", data=body, method="PATCH", headers=hdr)
with ur.urlopen(req) as r:
    pass

# 4) 加 block (PATCH 父)
body = json.dumps({"children": [new_block]}).encode()
req = ur.Request(f"https://api.notion.com/v1/blocks/{page_id}/children", data=body, method="PATCH", headers=hdr)
with ur.urlopen(req) as r:
    pass
```

**重要**：引号嵌套要小心（外双内单，或外单内双），中文引号 `"…"` 也要统一。

### 1.8 重检铁律清单的 4 类问题（实战踩过的坑）

| 问题类型 | 症状 | 修复 |
|---|---|---|
| 标题条数与实际不符 | 标题"11 条"但表里实际 10 条 | 重数一遍 · 4+3+4=11 ≠ 标题"10 条" 之类 |
| 分组没标条数 | "范式路由" 没说几条 | 补"4 条"/"3 条"/"4 条" |
| 长解释混在铁律表 | "#90 ...（8s < x ≤ 15s 完全合法...）" | 解释性文字 → 移到 references · 铁律表只留短句 |
| 末尾指针错误 | 末尾"反 v15 范式铁律 = 7 条保留" 指向不存在的"7 条" | 改实际条数 · 或指向具体下方表名 |

## Part 2 · v7.1 混合合并范式（Pic8 Rabbit 实战沉淀 · 2026-06-10）

### 2.1 触发条件

绘本走 v7 范式（5 条件过）**但**有**高密度段**（家族词组集合 ≥ 3 词 / 全文+微动画 / 强文字持续可见）→ **v7.1 混合合并范式**。

**不是 v7（2图=1Clip 全合并）或 v15（1图=1Clip 全独立），而是混合**。

### 2.2 判定 3 问

| 问 | 答案 → 走法 |
|---|---|
| 1. 普通段（旁白短+单焦点）有多少？ | ≥ 3 段 → 普通段走 v7 2图=1Clip 合并 |
| 2. 有高密度段（家族词组/全文+微动画）吗？ | 有 → 独立 Clip（不合并） |
| 3. 总图数 ≤ 10 张吗？ | 是 → v7.1 甜区 · >10 张走 v15 |

### 2.3 时长公式（Pic8 Rabbit 5 Clip 范本）

| 段类型 | 时长 | Pic8 Rabbit 实测 |
|---|---|---|
| 普通段 1（旁白短+单焦点）| 11s | Clip1=11s（旁白 1+2 · 图 1→图 2）|
| 普通段 2 | 12s | Clip2=12s（旁白 3+4 · 图 3→图 4）|
| 普通段 3 | 12s | Clip3=12s（旁白 5+6 · 图 5→图 6）|
| **高密度段**（家族词组/全文+微动画）| **14s 起步** | **Clip4=14s（旁白 7 IT 家族 5 词 · 图 7）** ⭐ |
| 收势/封面段 | 6-8s | Clip5=6s（旁白 8 收势 · 图 8）|
| **总时长** | **55s** | 5 Clip 实战范本 |

### 2.4 关键判定口诀

> **"单段内容密度高 = 独立 Clip + 14s；普通段 = 合并"**

### 2.5 4 反模式（Pic8 Rabbit 翻车沉淀）

- ❌ **5 词家族塞 2图=1Clip 合并**（11-12s 装不下 5 词+微动画+静默 = 末帧 < 2s 翻车征兆）
- ❌ **给家族段 8s 上限**（误读 v5 默认档 5/6/7/8s 为硬上限 = 实际 8-15s 都合法）
- ❌ **用 v15 范式填 v7 合并**（v15 模板只支持单图 @ImageN，不支持 @Image1 + @Image2 双图）
- ❌ **全合并或全独立的二选一思维**（v7.1 混合是正解）

### 2.6 必填 seedance 参数

```bash
python3 seedance.py create \
  --image <段首图> \
  --last-frame <段尾图> \
  --prompt "<v7 8 段 prompt 文本 + 高密度段特化>" \
  --duration 8/8/9/10/14 \
  --ratio 16:9 \
  --generate-audio <true|false> \  # 家族词组 = false（铁律 #86）
  --watermark false \
  --download
```
