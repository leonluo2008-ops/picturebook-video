# 飞书视频交付与目检反例排查手册（2026-06-07 沉淀）

> **背景**：Pic4 No 不绘本 clip1 跑通后，vision_analyze × 3 帧 + mcp_zai_analyze_video × 1 次 = 4 次独立验证**全部确认视频是 No 内容**。但用户**目检反馈"看到的是 Welcome"**。**根因未实锤**。
>
> **本手册目的**：未来面对"用户目检 vs vision_analyze 不一致"的情况，**有一套标准化排查 + 应对流程**，不必每次重新摸索。

## 4 次独立验证都说什么

```
vision_analyze frame_t0.5 → "小熊 Stop 拒绝手势 + 顶部 No/不能 文字"
vision_analyze frame_t2.0 → "小熊双掌举到脸颊两侧 + 文字保留"
vision_analyze frame_t3.5 → "小熊 Stop 拒绝手势 + 末帧微动"
mcp_zai_analyze_video → "是 No绘本不是 Welcome绘本"
```

## 4 个排查事实点（必查）

```
1. 本地图 = 期望绘本？       → vision_analyze 一帧
2. chevereto 上传 = 本地图？  → md5 比对
3. pic4 mp4 md5 ≠ pic3 mp4 md5？  → md5 比对（排除发错文件）
4. mp4 内容 = 期望？        → vision_analyze 多帧 + mcp_zai_analyze_video
```

**Pic4 排查结果**：4 个事实点**全部指向视频是 No 内容**。

## 4 个可能根因（按概率从高到低）

| 概率 | 根因 | 证据 | 验证方法 |
|---|---|---|---|
| **高** | A. 飞书 client 渲染 bug（附件显示成历史消息）| 用户目检 = 历史消息内容 | 用户**长按附件查看原文件** + md5 验证 |
| **中** | B. 用户看到的是历史 pic3 9 段消息中的某条 Welcome 视频| 今天发过 pic3 9 个 + pic4 1 个 = 10 个视频 | 用户看发送时间 + 文件名 |
| **中** | C. seedance 实际生成跟 prompt 不一致（vision 看花眼）| vision × 4 全部确认是 No = 不太可能但不能 100% 排除 | 重新跑一次 + 用户目检 |
| **低** | D. mp4 → 飞书附件 → 用户手机 = 渲染链不可控（codec 不兼容）| 罕见但发生过 | 用户换客户端或本地播放器打开 |

## 修复方向（铁律 #63 + 铁律 #65）

### 1. 主 agent 每次发飞书**附完整证据链**

```markdown
📹 [绘本] clip[N] 视频
- task_id: cgt-...
- 文件: v1-clip[N]-fixed.mp4
- md5: ...
- 原图: [MEDIA:1.jpg] md5 ...
- prompt 头部: "主体定义：..."

请你：
1. 长按视频查看原文件（不要看缩略图）
2. 比对 md5
3. 看视频内容是否是 [绘本主题]
```

### 2. 视频交付不只发飞书，**同时落地磁盘 + 飞书云盘**

- 本地磁盘：`/home/luo/huiben-projects/<日期-项目>/v1-clip[N]-fixed.mp4`（永远有）
- 飞书云盘：上传到对应文件夹（用户可从云盘二次下载验证）
- 飞书消息：附 MEDIA（**但预期可能渲染失败**）

### 3. **vision_analyze 跟人眼观感不一致时，以人眼为准**

- vision 是**辅助**不是**真理**
- 多个 vision 验证 + mcp_zai_analyze_video = 高置信度，但**不是 100%**
- **人眼是金标准**——用户说"看到的是 X" = X，不能凭 vision 反驳

### 4. **单 Clip 端到端验证 = 等用户目检 = 不批量跑**

**反模式**（Pic4 v1.0.2 差点触发）：
- 单 Clip 跑通 → vision × 3 帧全过 → "应该没问题" → 直接批量跑剩下 8 段
- 结果：用户反馈"内容不对" → 已经跑完 8 段 → 浪费 ~16 min + 8 个视频产物

**正确做法**：
- 单 Clip 跑通 → 发飞书 + 完整证据链 → **等用户目检确认** → 用户说"OK 继续"才批量
- 用户目检发现"内容不对" → 先排查再决定批量
- **D 跑 8 段 = 不可逆成本**（不是临时计算），必先 1 段目检

## 排查话术模板（用户目检发现不一致时）

```markdown
我排查了 4 个事实点：
1. ✅ 本地 1.jpg = No 绘本（vision 确认）
2. ✅ chevereto 上传的就是本地 1.jpg（md5 一致）
3. ✅ pic4 v1-clip1-fixed.mp4 md5 跟 pic3 v1-clip1-fixed.mp4 md5 不同
4. ✅ vision_analyze 多个工具看 pic4 视频**确认是 No 内容**

可能根因（未实锤）：
- A: 飞书 client 渲染 bug（附件显示成历史消息）— 概率高
- B: 你看到的是 pic3 历史消息里的某条 Welcome 视频 — 概率中
- C: seedance 实际生成跟 prompt 不一致 — 概率低
- D: mp4 渲染链不可控 — 概率低

请你：
1. 长按附件查看原文件
2. 比对 md5
3. 看视频内容

如果是 No = 之前你看的是历史消息
如果是 Welcome = 飞书渲染 bug / 我有别的隐藏问题
```

## 排查流程（决策树）

```
用户目检发现 "内容是 X 不是 Y"
   ↓
1. 验证本地图 = Y?
   ↓ 否 → 修本地图 → 重新跑
   ↓ 是
2. 验证 chevereto 上传 = 本地图? (md5 比对)
   ↓ 否 → 修 chevereto 上传 → 重新跑
   ↓ 是
3. 验证 mp4 md5 = 期望 mp4 md5? (排除发错文件)
   ↓ 否 → 修附件发送 → 重新发
   ↓ 是
4. vision_analyze mp4 多帧 = Y?
   ↓ 否 → seedance 实际生成错 → 改 prompt → 重新跑
   ↓ 是
5. mcp_zai_analyze_video mp4 = Y?
   ↓ 否 → vision 抽帧有偏 → 接受 vision 结论 → 重新发
   ↓ 是
6. 4 步全过 = 视频内容确认是 Y
   ↓
7. 用户目检说 X = 飞书渲染 / 用户看错历史
   ↓
8. 重新发 + 完整证据链（让用户能 100% 验证）
   ↓
9. 用户说 "确认是 Y" = 之前看错 = 继续批量
   ↓ 否 = 飞书渲染 bug = 落地磁盘 + 云盘 + 接受飞书渲染失败
```
