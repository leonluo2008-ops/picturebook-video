## 1. 元教训 #10 · 不把"建议"升级成"红线"

### 1.1 翻车现场

我之前在 SKILL.md / reference 写了：

> "**模型对精确时间（如 0–3 秒）的支持不稳定，强行限制时长可能导致生成结果异常**"——**这条对我们之前的 v15"按秒数硬切分镜"是直接打脸**

我把官方原话**包了引号**包装成"官方依据"强推给用户——**还升级成 v15"按秒数硬切"是错的**的判断。

### 1.2 用户的纠错

> "**模型对时间精确是很稳定的，很准的。这条资料肯定不是官方文档里的吧？**"
> "**官方的意思只是不要在同一个时间点强行塞入很多镜头和动作……你可以按照官方的这种做法，在我们写提示词的时候来进行调整，比如说我们现在用的就是精确控制时间，如果发现精确控制时间不行，那我们就用镜头**"

### 1.3 真实官方原话（doc2 §2 · "使用分镜时序" 章节末小字）

> "使用`镜头1`、`镜头2`、`镜头3`等标识，按事件发生顺序（先主后次）组织内容。**不强制限制每段时长，优先让模型根据剧情自然生成节奏。**"
>
> "**说明** 模型对精确时间（如 0–3 秒）的支持不稳定，强行限制时长可能导致生成结果异常。"

### 1.4 4 个语气等级（区分清楚）

官方文档的语气分 4 档：

| 原文位置 | 语气 | 处理 |
|---|---|---|
| **正文**（"提示词进阶公式 = 精准主体 + 动作细节 + ..."）| **必填** | 必写 |
| **红框 / 加粗 / "注意" 框**（"注意 一个镜头里尽量只指定 1 种运镜方式..."）| **红线** | 必避 |
| **小字 / "说明" 框**（"说明 模型对精确时间支持不稳定..."）| **建议** | 优先用，不当铁律 |
| **章节"案例" / "其他技巧"** | **示例** | 灵感来源，不当模板 |

### 1.5 反模式 vs 正解

```markdown
# ❌ 错：把"说明"读成"红线"
"官方明确说'按秒数拆是错的'——v15 必改成'按事件拆'"（过度推论）

# ✅ 对：把"说明"当"建议"
"官方建议'按事件拆'优先——v15 当前'按秒数拆'也是合法选项，实战翻车再切回'按事件拆'"（两种都行）
```

### 1.6 修复判断口诀

> **"建议 ≠ 红线 · '可能' ≠ '必然' · '不强制' ≠ '禁止'"**
>
> 引用官方原话前必先回原文档 grep 验证原文 + 标注上下文（"建议"/"说明"/"注意"/"红线"）。

## 2. 元教训 #11 · 写 prompt 前 = 必先 vision 参考图 + 列"图里有/没有"清单

### 2.1 翻车现场

绘本 clip1 v1 v2 翻车的元问题 = **我没用 vision 看参考图**就写 prompt：

| 我编的 | 参考图里有没有 |
|---|---|
| "yellow duck family" | ❌ **3 张图都是 1 只鸭**，没有"family" |
| "1 large duck + 3 small ducklings on green lily pads" | ❌ **没有小鸭，没有荷叶** |
| "大鸭子轻轻歪头"（v2 凭空加） | ❌ **3 张图鸭子都没歪头** |
| "水花声"（v1 凭空加）| ❌ 图 1 草地 / 图 2 静水（无水花）|

**我完全没看参考图**——靠"v0 的绘本应该长这样"的印象 + 抄 v1 prompt 编了 4 个核心元素。

### 2.2 用户的纠错

> "**你在写第二版提示词时没有考虑到参考图，写提示词时要预测参考图画面接下来可能发生的事情，根据推测顺着画面发展的方向去写，而不是凭空捏造。这样我看了根本和参考图对不上，导致画面很混乱，连唯一的效果都达不到。**"

### 2.3 正解：写前 3 步

```python
# Step 1：vision 全 N 张图（不抽样）
for img in image_list:
    vision_analyze(img)  # 1 张都别跳

# Step 2：列"图里有/没有"清单（写到 image-inventory.md）
# 例：绘本 clip1
inventory = {
    "图1": {"有": ["1 只黄色大鸭", "草地背景", "纸艺拼贴"], "没有": ["小鸭", "荷叶", "水"]},
    "图2": {"有": ["1 只黄色大鸭", "水面+静态蓝圈波纹"], "没有": ["小鸭", "荷叶", "水花"]},
    "图3": {"有": ["1 只黄色大鸭", "踏浪/奔跑姿态", "蓝色水花"], "没有": ["小鸭", "荷叶"]},
}

# Step 3：每个镜头 = 显式 @ImageN 绑定 + 只用清单里"有"的元素
"镜头 1：参考 @Image1，一只黄色鸭子站在草地上..."  # ✅ 草地图1有·"草地"在清单里
"镜头 1：参考 @Image1，4 鸭立荷叶..."  # ❌ 荷叶不在清单里 = 凭空捏造
```

### 2.4 关键设计

- **每镜头显式 `@ImageN`** 绑定到具体那张图（不写"3 镜头 = 3 张图" 的死规则——可以复用 #13）
- **只用清单里"有"的元素**——清单里没的元素 = 凭空捏造 = 必删
- **"没有" = 关键警告**——写下"图里没有 X" 防止 prompt 误写

### 2.5 自检清单（写完 prompt 必跑）

| # | 检查 | 期望 |
|---|---|---|
| 1 | prompt 里**每个**名词都在 image-inventory 的"有"列里？| ✅ 0 凭空捏造 |
| 2 | prompt 里**每个**"有"列元素都被某镜头引用了？| ✅ 0 浪费 |
| 3 | prompt 里**没有**"没有"列元素？| ✅ 0 矛盾 |

## 3. 元教训 #12 · 凡引用官方原文 = 必逐字核对（漏字 = 根因）

### 3.1 翻车现场

我之前把官方 §4 红线"**一个镜头里尽量只指定 1 种运镜方式**"读成：

> "**1 镜 1 动作**"

漏了"**运镜方式**"4 个字——**4 个字的漏字**把"运镜"读成"动作"——直接让我建议用户"1 镜只 1 动作"——**违反官方愿意**（"1 镜多动作 OK"）。

### 3.2 用户的纠错

> "**我认为你对官方关于镜头和动作的理解有问题。你说一个镜头对应一个动作这是绝对不行的，我可以明确告诉你。官方的意思只是不要在同一个时间点强行塞入很多镜头和动作，这可能会造成冲突，但不是说在一个镜头里只能使用一个动作。比如在开场的一点五秒镜头里，你可以使用一个镜头配合三个动作：鸭子往左走，然后飞起来，这些都是可以的。**"

### 3.3 真实官方原话（doc2 §4 · "运镜写法" 章节末红框）

> "**注意 一个镜头里尽量只指定 1 种运镜方式，不要同时要求推拉摇移，会增加画面的不稳定性**"

漏 4 字 = 错全意：
- 漏"**运镜方式**"4 字 = "运镜"被读成"动作"= "1 镜 1 运镜"被读成"1 镜 1 动作"
- 漏"**注意**"红框 = "红线"被读成"建议"

### 3.4 引用官方原话 SOP（必走）

```python
# Step 1：定位原文档 + 原文（grep 验证）
content = open("official_doc.md").read()
matches = grep(r"一个镜头", content)  # 必用正则
print(matches.group(0))  # 完整原文

# Step 2：标注 4 维属性
quote_meta = {
    "原文": matches.group(0),  # 完整原句，不缩略
    "位置": "doc2 §4 line286-288",  # 文档+章节+行号
    "语气": "红框/加粗/'注意'",  # 4 档之一
    "上下文": "前后 3 句"  # 必带，防断章取义
}

# Step 3：禁止"凭记忆复述"
# 任何 "官方说 X" = 必带 4 维属性 = 必查原文档验证
```

### 3.5 反模式

```markdown
# ❌ 错：凭缩略记忆复述
"官方说 1 镜 1 动作"

# ❌ 错：断章取义
"官方支持 1 镜 3 动作"（只引 1 半）

# ✅ 对：原文 + 4 维属性
doc2 §4 line286-288（红框·"注意"语气）："**一个镜头里尽量只指定 1 种运镜方式**"——仅指运镜（推/拉/摇/移/切），动作可以多
```

### 3.6 元教训口诀

> **"漏 1 字 = 错全意 · 缩 1 句 = 断章取义 · 凭记忆 = 翻车"**

## 4. 元教训 #13 · 跳过有问题的参考图 = prompt 里也不提 + 复用其他参考图 + 描述"自然延伸"

### 4.1 翻车现场

绘本 clip1 v3 用 @Image3（踏浪跑步姿态）= 跟"鸭子游泳"主题矛盾 = 末帧虽然 v3 修了"冲浪"但 v3 末帧仍是"踏浪"姿态（不是游泳）——**不是修复 prompt 能修的**——**@Image3 这张图本身就跟主题冲突**。

### 4.2 用户的纠错

> "**@image3 本身就有问题……那我们遇到这种情况你就用参考图二，第三个镜头也用参考图二，继续让这只鸭子，比如说你在切镜头的时候，第三个镜头仍然用参考图二，你描述这只鸭子在水里游泳就可以了。那我们就不使用有问题的这张图**"

### 4.3 正解

| 情况 | 处理 |
|---|---|
| **参考图 N 跟主题不冲突** | prompt 引用 + 描述其姿态 |
| **参考图 N 跟主题冲突 / 姿态异常** | **prompt 里不引用** + 复用 @ImageM + 描述"自然延伸"（"参考 @ImageM 的同一只鸭子，镜头继续中景跟拍，鸭子从浮水状态开始游动"） |
| **所有参考图都跟主题冲突** | **不传参考图** + 用 `--prompt` 纯文字生成（v3 范式）|

### 4.4 关键技巧 · "镜头延续" 写法

复用 @ImageM 描述延伸动作时，**不重新描述姿态**——**用"延续"**：

```python
# ❌ 错：重新描述姿态（变成"另一只鸭子"）
"镜头 3：参考 @Image2，一只黄色鸭子开始向画面右侧游动，脚蹼在水面下划水，翅膀贴着身体不张开，水面泛起几圈涟漪"
# 问题：5 个限制词堆叠 = "约束词污染"（#14）

# ✅ 对：镜头延续（自然过渡）
"镜头 3：参考 @Image2 的同一只黄色鸭子，镜头继续中景跟拍，鸭子从浮水状态开始向画面右侧游动，水面泛起几圈涟漪"
# 关键："参考 @Image2 的同一只鸭子" + "镜头继续" + "从浮水状态开始" = 自然延伸
```

### 4.5 自检清单

| # | 检查 | 期望 |
|---|---|---|
| 1 | prompt 引用的 @ImageN = 跟主题不冲突的图？| ✅ 不引用冲突图 |
| 2 | 复用 @ImageM 时是否用"参考 @ImageM 的同一只" + "镜头延续" 写法？| ✅ 不是"另一只鸭子" |
| 3 | ref_images 列表里是否 0 张冲突图？| ✅ 0 冲突 |

## 5. 元教训 #14 · "不要 X 不要 X" = 约束词污染

### 5.1 翻车现场

我之前 v4 末尾约束：

> "**保持无字幕、无水印、无 Logo，无人声、无歌唱、无配音、无朗读。音效只保留环境细节：水花声、纸页翻动等短促物理声。全程无背景音乐。**"

**6 个"无 X"连用 + 2 个"音效只保留" + 1 个"全程无 BGM"**——**全是"不要 X"的否定式堆砌**。

### 5.2 用户的纠错

> "**我认为你的提示词写法里有一些问题，就是负面的不要什么不要什么不要什么，你根本就没有必要，你可以去看官方的资料，他们怎么说的提示词，怎么写？……这样反而会对模型造成污染**"

### 5.3 真实官方原话（doc2 §5 · "画质、风格与约束词"）

官方**只**列了 3 个"不要 X"模板：

```
- 避免生成字幕："保持无字幕"
- 避免生成 Logo："不要生成Logo"
- 避免生成水印："不要生成水印"
```

**全部是"针对已知会出错的特定元素"**——不是泛泛"不要 X"。

### 5.4 正解

| 元素 | 写法 | 出处 |
|---|---|---|
| **字幕** | "保持无字幕" | 官方原话 |
| **Logo** | "不要生成 Logo" | 官方原话 |
| **水印** | "不要生成水印" | 官方原话 |
| **人声/歌唱/配音/朗读** | "无人声、无歌唱、无配音、无朗读"（4 词）| 绘本专用（用户已确认保留）|
| **BGM** | 不写（默认走官方策略）| 用户红线"绘本不生成 BGM" |
| ~~"音效只保留环境细节"~~ | **不写** | ❌ 泛泛约束 = 污染 |
| ~~"全程无背景音乐"~~ | **不写** | ❌ 跟"无 BGM"重复 + 泛泛 |
| ~~"翅膀不张开" / "脚蹼在水下划" / "细小涟漪"~~ | **不写**（用"延续"写法替代）| ❌ "不要 X" 在镜头描述里 = 污染 |

### 5.5 反模式 vs 正解

```python
# ❌ 错：泛泛"不要 X"堆砌
"保持无字幕、无水印、无 Logo，无人声、无歌唱、无配音、无朗读。
音效只保留环境细节：水花声、纸页翻动等短促物理声。
全程无背景音乐。"

# ✅ 对：精简到官方 3 类 + 4 词
"保持无字幕。
不要生成水印。
不要生成 Logo。
无人声、无歌唱、无配音、无朗读。"
```

### 5.6 元教训口诀

> **"约束词只对'已知出错元素'用 · 泛泛'不要 X' = 污染"**
>
> 4 词末尾约束（无人声/无歌唱/无配音/无朗读）是绘本专用约束 = 已确认保留 · **不**再开问卷问"要不要删"

## 6. 元教训 #6（已存在的种子）· MCP timeout ≠ 任务失败

### 6.1 翻车现场

5 个版本（v1/v2/v3/v4/v5）每次 `wait_and_download` 都报 `MCP call timed out after 120.0s`——我每次都先慌一下，然后改用 `check_task` 查 status 发现是 running，再调 `wait_and_download` 等到 succeeded。

### 6.2 用户的元偏好（已在 memory）

> "**不要重跑前台，已提交的任务，只要有了 task id 就表示扣费，你检查 API 用法，使用 task id 去查询状态，并下载视频**"

### 6.3 正确流程（绘本单 Clip 场景）

```python
# Step 1：提交（不加 --wait，不前台阻塞）
TASK_ID = mcp_seedance_generate_video(...)  # 拿 task_id 立即返回

# Step 2：第一次轮询
result = mcp_seedance_wait_and_download(task_id, timeout=480)
# 可能 timeout（120s poll 上限）—— 正常

# Step 3：timeout ≠ 失败 —— 用 check_task 查 status
status = mcp_seedance_check_task(task_id)
# status = "running" | "queued" | "succeeded" | "failed"
# 任何非 succeeded → 等，再调 wait_and_download

# Step 4：succeeded 后的 output_path 就是本地文件路径
# 不需要单独 download（wait_and_download 已自动下载）
```

### 6.4 反模式

```python
# ❌ 反模式 1：看到 timeout 就重提交
mcp_seedance_wait_and_download(...)
# timeout → mcp_seedance_generate_video(...)  # ❌ 重复扣费

# ❌ 反模式 2：认为 timeout = 任务失败
# 实际是 seedance 14s 720p 视频生成需要 5-15 分钟，超过 MCP poll 120s 上限 = 正常
```
