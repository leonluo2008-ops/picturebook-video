# 跨平台分发文件名乱码（2026-06-16 I love you 我爱你 绘本踩坑）

## 现象（必读）

绘本/漫剧/短视频成品做跨平台分发时，**zip / 压缩包 / 飞书云盘 / 邮件附件** 的文件名含中文（UTF-8 编码）→ **Windows 资源管理器按 GBK 解码** = 乱码（"我爱你" → "ÎÒ°®Äã"）→ 用户打开 zip 看到文件名乱码 = 体验差。

## 根因

zip 格式是**无 BOM 标记**的二进制格式：文件名按创建时的 locale 编码写入，**跨 locale 时按接收端默认编码解码**。

- Linux 默认 UTF-8 locale → 文件名按 UTF-8 写入 = 正确
- Windows 默认 GBK locale（中文系统）→ 收到 zip 时按 GBK 解码 UTF-8 字节 = **乱码**
- macOS Finder 旧版本同坑（10.x 早期）

**核心矛盾**：UTF-8 中文文件名**无法**通过 zip 这种"无 BOM 标记"格式保证 Windows 100% 显示正确。

## 修复范式（3 条 · 必走）

### 方案 A · 纯英文/数字文件名（最稳 · 100% 兼容）

```
✅ clip01.mp4 / clip02.mp4 / clip03.mp4 ...
❌ 01-亮相-I_LOVE_YOU.mp4 （含中文）
```

适用场景：默认首选 · 任何跨平台分发 · 跨 Windows/Mac/Linux/iOS/Android 全平台零乱码。

### 方案 B · 拼音文件名（次优 · 仍 UTF-8 风险但能拼读音）

```
✅ 01-xianliang-I_LOVE_YOU.mp4
❌ 01-亮相-I_LOVE_YOU.mp4
```

适用场景：需要可读性 + 跨平台 · 拼音在 UTF-8 字节下不会像汉字那样严重乱码（至少能拼出读音）。

### 方案 C · 英文描述文件名（兼顾可读性 + 100% 兼容）

```
✅ 01-Opening-I_LOVE_YOU.mp4
❌ 01-亮相-I_LOVE_YOU.mp4
```

适用场景：需要看文件名识别角色 · 跨 Windows/Mac/Linux 100% 兼容。

## 3 方案对比

| 方案 | 中文可见性 | 跨平台兼容性 | 推荐度 |
|---|---|---|---|
| A · 纯英文/数字 | ❌ 看不懂中文 | ✅ 100% 兼容 | ⭐⭐⭐⭐ |
| B · 拼音 | ⚠️ 看不懂中文但能识别角色 | ✅ 100% 兼容 | ⭐⭐⭐ |
| C · 英文描述 | ❌ 看不到中文 | ✅ 100% 兼容 | ⭐⭐⭐⭐ |

## 反模式（必避）

- ❌ zip 内部用中文名 + 飞书消息里也用中文名 = 用户两边都看到乱码
- ❌ 默认走"中英混合名"（"01-亮相-I_LOVE_YOU.mp4"）= Windows 必乱码
- ❌ 寄希望于"Windows 11 24H2 修了 zip 编码" = 没修 + 用户系统版本不一 = 不可靠
- ❌ 改用 rar/7z 格式绕过 = 用户不一定装了 = 体验差
- ✅ 纯英文/数字命名 + 中文名只放飞书消息正文里

## 中文名放哪里

**zip 内部**：纯英文/数字命名（方案 A 默认）

**飞书消息正文**（发出去时）：可以写中文角色表 + 序号映射：

```
文件清单：
- clip01.mp4 = 亮相页（粉红+橘猫+爱心）
- clip02.mp4 = 父母场景（室内 Mom + 户外 Dad）
- clip03.mp4 = 祖辈场景（奶奶 + 爷爷）
- clip04.mp4 = 朋友 + 小鸟场景
- clip05.mp4 = 太阳场景
- clip06.mp4 = 收势我爱你页
```

**这样设计的好处**：
- zip 内部文件名 = 跨平台 100% 兼容（无乱码风险）
- 中文角色表 = 飞书消息里能正常显示（UTF-8 通道）
- 用户两边都看到正确信息

## 实测案例（2026-06-16 I love you 我爱你 绘本）

**第一次打包**（用方案 B 拼音版）：
- 文件名：`01-xianliang-I_LOVE_YOU.mp4` / `02-fumu-Mom_Dad.mp4` / ... 
- 用户反馈："重命名里的中文乱码，请解决乱码问题"
- **根因**：拼音版没用，但含中文括号备注"重命名里的中文" = 用户**实际看的是 zip 内部的`clips/01-亮相-...mp4`路径** = 中文乱码（zsh 截图显示 `clips/01-亮相-I_LOVE_YOU.mp4`）

**第二次打包**（用方案 A 纯数字版）：
- 文件名：`clips/clip01.mp4` / `clips/clip02.mp4` / ... 
- 用户确认：OK · 零乱码

**根因定位**：第一次打包的 zip **内部路径**是 `clips/01-亮相-...mp4`（中文乱码）= 不是文件名问题，是 **zip entry path** 含中文 = 走 GBK 乱码。

**修复 2.0**：zip entry 路径**也必须纯英文/数字**（不是 zip 压缩包名）= `clips/clip01.mp4` 而非 `clips/01-亮相-...mp4`。

## 关键修复点（必走）

1. **zip 压缩包名**：可中文（如 `I_love_you_我爱你_6clips.zip`，飞书云盘 UTF-8 通道正常）
2. **zip 内部 entry 路径**：**必纯英文/数字**（`clips/clip01.mp4`）
3. **mp4 文件名**：**必纯英文/数字**（`clip01.mp4`）
4. **飞书消息正文**：写中文角色表（让用户能识别哪个 clip 是哪个场景）

**判定口诀**：**"压缩包可中文 · entry 路径必纯英文 · 文件名必纯英文 · 消息正文写中文"**。

## 跟 zip 完整性验证的配合

```bash
# 1. 验证 zip 完整性（必跑）
unzip -t my_zips.zip 2>&1 | tail -3
# 期望：No errors detected in compressed data

# 2. 列出 zip 内容（确认 entry 路径都是纯英文）
unzip -l my_zips.zip
# 期望：每个 entry 路径都是 ASCII 字符（无中文）
```
