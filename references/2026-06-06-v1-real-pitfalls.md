# v1.0.0 实战 Pitfall 库（Pic2 8 Clip 实测沉淀）

> **目的**：沉淀 v1.0.0 多 agent 架构在 Pic2 8 Clip 实战中发现的 pitfall，供 C/D 子 agent 后续调用参考。
>
> **来源**：2026-06-06 Pic2 Please 请绘本 8 Clip 端到端实战（A→B→C→D 全链路跑通）
>
> **方法**：实战翻车 = C 子 agent self_check 9/9 但 seedance 实际跑出 5/6 false → 暴露 C 子 agent 只查文本合规不查实际输出的盲区

## 实战总览

| 阶段 | 状态 | 翻车征兆 |
|---|---|---|
| A 风格识别 | ✅ 通过 | tone=活泼 / rhythm=快 / style_anchor 命中 |
| B 旁白量化 | ✅ 通过 | 兜底公式（用户没给 TTS）|

**核心矛盾**：**C 子 agent self_check 通过 ≠ 视频实际通过**——seedance 跑出来的结果 C 子 agent 看不到。

## Pitfall #1 · 画面在配音（反 Cat v15 范式）

**触发场景**：镜头三写"嘴巴半张开做说话口型礼貌地说出 X"

**反 Cat v15 范式**：画面应该先讲语义，朗读只承担读音强化

**修复**：
- 镜头三写法改为"画面呈现<clip_narrative>故事动作"
- 必填字段 `clip_narrative`（1 句话描述本 Clip 故事动作）
- Pic2 Clip 1 = "妈妈翻开书页 + 小兔子眼睛盯着书"
- Pic2 Clip 8 = "兔子竖大拇指 + 爱心飘出 + 夕阳渐变"

**Cat v15.1 实战对照**（assets/example-prompts/cat-clips-1-6-v15.1.txt）：
- pat Clip：画面 = 猫掌按地 → 朗读强化"pat"读音
- cat Clip：画面 = 小猫抬头 → 朗读强化"cat"读音
- **不是**"小猫嘴巴半张开说 pat"

**根因**：v1.0.0 重构时只借了 v15 范式的 4 段骨架 + 节奏公式，**漏借了"画面先讲语义"的核心约束**。

## Pitfall #2 · 短档镜头一 1s 不够时间

**触发场景**：节奏 1-1-3-1 短档（旁白 < 4s 朗读），镜头一 1s 写"切到全景"

**翻车征兆**：seedance 跑出 frame_01 角色完全丢失（"切"动作本身吃 0.5s，留给主体出现的窗口只有 0.5s）

**修复**：
- 短档节奏 1-1-3-1（6s）→ **1.5-1-3-1.5（7s）**
- 镜头一 1s → **1.5s**（参考 v15 长档 2s 的"足够时间"哲学，按比例缩 0.5s）
- 末帧 1s → 1.5s（同样理由）

**Cat v15.1 实战对照**：
- Cat 6 Clip 全部用 v15.1 长档 12s（节奏 2-1-3-3-3-3-3）
- 镜头一 2s = 拉远足够时间
- v1.0.0 短档 1s = 凭印象"短就短"——没参考 v15 长档 2s 的"足够时间"哲学

**根因**：节奏档位表只列了时长，没列"镜头一最少时间"

## Pitfall #3 · C 子 agent self_check 只查文本合规（不查实际输出）

**触发场景**：C 子 agent 输出 prompt_draft 后 self_check 9/9 通过，但 seedance 跑出来 5/6 false

**盲区**：
- C 子 agent self_check 9 项**只检查 prompt 文本**：@Image 语法 / 角色指代 / 无隔离句 / 末帧文字 / 节奏档位等
- **不检查** seedance 实际跑出来是否：① 主体完整 ② 文字完整保留 ③ 末帧画面微动 ④ 4 镜头对齐

**修复**：
- C 子 agent 必填 `clip_narrative` 字段（Pitfall #1 修复）—— **不**仅是文本约束，是"让画面讲什么"的具体描述
- C 子 agent 必填 `seedance_visual_checklist` 字段 —— 描述 D 子 agent 跑完后必看的视觉点（如"frame_01 主体完整 + frame_05 文字仍在 + frame_06 末帧微动"）
- D 子 agent 实战 self_check 6 项（已在 D SKILL.md 列出）—— **必报回主 agent**（铁律 #45）
- 主 agent 翻车决策树：实战翻车 → **重发 C 改 prompt**（不重跑 D）

**根因**：v1.0.0 多 agent 架构 C/D 边界是"设计 vs 执行"，但 C 只看设计文本不看实际效果——**缺"反馈环"**

**未来修复方向**（v1.1 待办）：
- C 子 agent 加 evals 跑测（pic 已知翻车征兆 + 自动复跑验证）
- 或：D 子 agent 跑完反馈给 C 子 agent 自校（C 看实际效果调整 prompt）
- 当前 v1.0.0：**人工 review 视频 → 手动重发 C**（已 work）

## Pitfall #4 · 实战翻车不擅自重跑 D（铁律 #45）

**触发场景**：D 子 agent 跑完视频 → 抽帧自检 5/6 false

**正确处理**：
- D 子 agent **不擅自改 prompt 重跑**（铁律 #45）
- D 子 agent 把 `is_flipped: true` + `flip_reasons: [...]` 报回主 agent
- 主 agent 看到翻车 → 决策"重发 C 改 prompt"（不重跑 D）

**反模式**：
- D 子 agent 看到"文字消失"擅自把 prompt 改强 → 翻车扩散（影响面更大）
- 主 agent 看到翻车直接重跑 D 同一 prompt → 翻车征兆不变

**Pic2 实战**：D 子 agent 严格遵守边界 = 翻车 5/6 但报告完整 → 主 agent 决策"重发 C"→ 这是 v1.0.0 架构的胜利

**根因**：v1.0.0 边界清晰 = 翻车可控；边界混乱 = 翻车扩散

## Pitfall #5 · 未实战验证不能开 PR（铁律 #49）

**触发场景**：v1.0.0 重构完成（4 子 agent + 调度层 + evals 100% 命中）→ 主 agent 擅自开 PR

**用户原话**（2026-06-06）："现在肯定不能跑PR，我们都没有实测过，怎么能PR"

**正确处理**：
- v1.0.0 重构完成 + evals 100% 命中 → **不**自动开 PR
- 必须**实战验证**（跑 1-2 个真实绘本 mp4 视频）→ 主 agent 看完视频 → 主 agent 拍板 → 才能开 PR
- 实战翻车 → 修补 v1.0.0 → 重新实战验证 → 循环

**反模式**：
- "evals 全 100% 命中 = 实战必通过"（**错**——evals 只测文本不测视频）
- "重构完成 = 应该开 PR"（**错**——重构完成 ≠ 实战可用）
- "用户没明说不能开 PR = 默认开"（**错**——决策权默认归主 agent，但不可逆操作必问）

**根因**：evals 是 prompt 草稿质量量化，**实战是视频质量量化**——两个不同维度，**实测胜于 evals**

**Pic2 实战教训**：evals 100% → 实战 5/6 false = evals 只能保证"prompt 写对"不能保证"视频跑对"

## Pitfall #6 · 视频交付不抽帧（铁律 #29 强化）

**触发场景**：D 子 agent 跑完 8 个视频 → 主 agent 想发飞书给用户

**用户原话**（2026-06-06 实战末尾）："不要发抽帧，只发视频"

**正确处理**：
- 发飞书**只发视频**（[MEDIA: ...] 标签）
- **不**发抽帧图（[MEDIA: ...png] 标签）
- **不**主动抽帧验证（除非用户明确要求）

**反模式**：
- 发视频 + 抽 1 帧当预览（**用户零容忍**）
- "为了让你看效果抽 1 帧吧"（**自作主张**）

**根因**：用户原话零容忍 + 抽帧 = 浪费用户时间

**已沉淀**：D 子 agent SKILL.md 红线 #6 + 主 SKILL.md 视频交付流程
