## 4. 修复方向 1：节奏公式 = 动作成本（铁律 #52）

**用户原话**：
> **"两秒就把话讲完了，然后我们增加两秒让读者去消化，那剩下那两秒是不是把节奏给拖慢了呢？请思考这个背后的逻辑。"**

**根因**：v1.0.0 旧档 1-1-3-1（6s）套"Please, Mama, read!"（朗读 ~2.3s）= **凭空加 3.7s 空镜 = 节奏拖慢**。

**修复**：C 子 agent 节奏档位表重写：

| 旁白朗读 | 旧档 | 新档（动作成本）| 总时长 |
|---|---|---|---|
| < 1.5s | （未定义）| **极短档 1-朗读-1** | 3-4s |
| 1.5-3s | 1-1-3-1（6s）| **短句档 1-1-朗读-1** | 4-6s |
| 3-5s | （未定义）| **中句档 2-1-朗读-消化-收势** | 7-10s |
| 5-8s | 2-1-3-5（11s）| 长句档 2-1-3-消化-收势 | 10-15s |

**Pic2 实测**：总时长从 53s（v1.0.0 旧档）压到 48s（v1.0.0 新档）= 节省 5s（10%）。

## 5. 修复方向 2：Cat v15 范式精准 4 点（铁律 #53）

**用户原话**：
> **"我说的是对画面的镜头控制，画面的内容控制，特别是对clip做拆分的时候，那种控制，非常精准"**

**根因**：C 子 agent prompt 写"小兔子嘴巴半张开做说话口型礼貌地说出 Please" = **画面在配音**，不是**画面先讲语义**。

**修复**：C 子 agent 新增 4 条 Cat v15 范式精准约束 + 必填字段 `clip_narrative`：


## 7. 实战 gate 5 步（铁律 #51 重构/大版本发布前必跑）

```
1. 跑 1 个 Clip 端到端（验证流程跑通）
2. 看视频：末帧是否画面微动？文字是否保留？角色是否完整？
3. 跑 1-2 个完整绘本（8-12 个 Clip 批量）出 mp4
4. 抽 2-3 个 Clip 抽帧自检（不只凭感觉）
5. 全部 OK → 才能开 PR
```

**实战 OK 定义**：
- 至少 1 个 Clip 跑通端到端（A+B+C+D 全跑）
- 至少 1 个 Clip 视频自检 self_check ≥5/6
- 至少 1 个 Clip 末帧画面**真的**微动（不是定格海报）
- 至少 1 个 Clip 文字**真的**完整保留（不是消失）
